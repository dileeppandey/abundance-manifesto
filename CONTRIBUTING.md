# Contributing to the Abundance Manifesto

This repository grows through contributions. Every addition — an alternative tool, a translation, a signature — is a small token of abundance added to the commons.

There are four ways to contribute:

---

## 1. Sign the Pledge

Read [PLEDGE.md](PLEDGE.md) and open a Pull Request adding your name to [SIGNATORIES.md](SIGNATORIES.md).

This is the most personal contribution. It takes five minutes and lasts permanently.

---

## 2. Add an Open Alternative

The [alternatives directory](alternatives/) is a curated map of open-source and cooperative alternatives to proprietary products. It is the living proof that the manifesto's principles work.

### What makes a good entry

A good alternative entry answers four questions:
- What proprietary product does it replace?
- Why does it matter for abundance? (connect it to a manifesto principle)
- How mature is it? (be honest — a great early-stage project is still worth listing)
- How can someone get involved?

### Format

Each entry should follow this template:

```markdown
### [Tool Name](https://link-to-project.org)
**Replaces:** Proprietary Product Name  
**Principle:** [I] Radical Openness  
**Maturity:** Production-ready / Maturing / Early-stage  

One to three sentences describing what it does, what makes it distinctly open or cooperative, and what gap it fills. Be concrete — mention the license, the governance model, or a specific capability that matters.

**Get involved:** [Contribute upstream](https://github.com/...) · [Community forum](https://...)
```

### Maturity definitions

- **Production-ready** — used in production by real organizations; has active maintainers; documented
- **Maturing** — functional, actively developed, growing community, some rough edges
- **Early-stage** — promising but incomplete; worth watching and supporting

### To submit

1. Fork this repository
2. Add your entry to the appropriate file in `alternatives/`
3. Open a Pull Request titled `Add alternative: [Tool Name]`
4. In your PR description, briefly explain why this tool belongs here

If you're unsure which category fits, pick the closest one and note it in your PR.

---

## 3. Translate the Manifesto

The manifesto should be readable in every language. Translations live in `translations/`.

Name your file using the [ISO 639-1 language code](https://en.wikipedia.org/wiki/List_of_ISO_639-1_codes): `MANIFESTO_es.md` for Spanish, `MANIFESTO_fr.md` for French, etc.

Open a PR titled `Translation: [Language name]`.

---

## 4. Improve What's Here

- Fix errors, broken links, or outdated information
- Improve clarity or accessibility of the manifesto text
- Add context to existing alternatives
- Propose new alternative categories via an Issue

For substantive changes to the manifesto text itself, open an Issue first to discuss before editing.

---

## Standards

**Be accurate.** Don't list a project as "production-ready" if it isn't. Don't describe features it doesn't have. Honest assessments build trust.

**Be generous.** This is not a competition between open-source tools. If two tools serve similar purposes, list both with honest notes on their differences.

**Connect to the principles.** Every entry should link, even briefly, to one of the five manifesto principles. This keeps the directory an argument, not just a list.

**No ads, no hype.** Entries should describe what a tool is and does, not market it. Leave superlatives out.

---

## Recognition

Every contributor is listed in the GitHub commit history permanently. Significant contributors — translators, category curators, prolific alternative researchers — are recognized in a `CONTRIBUTORS.md` file.

If you maintain a specific category over time, you can be listed as its curator.

---

## Code of Conduct

This community is built on the belief that abundance requires cooperation. Be direct, be honest, and treat contributors with the respect you'd want in return. Dismissiveness, bad faith, and harassment have no place here.

Issues and PRs that violate this will be closed.
